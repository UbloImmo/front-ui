## Typing

In order to make the List component type safe while still being flexible, we needed it to be _generic_.

That means you need to define a type for the items you want to display.

> _We'll be using this `Pokemon` type in the examples below._

```typescript
type Pokemon = {
  id: number;
  name: string;
  base_experience: number;
  is_default: boolean;
  order: number;
  weight: number;
  power: number;
  attack: number;
  types: {
    slot: number;
    type: {
      id: number;
      name: string;
    };
  }[];
};
```


## Data Provider

The component expects a `dataProvider` hook to be provided.
**It is responsible for fetching / refetching, filtering and sorting the data.**
Your custom hook will the be ran internally, and therefore drive what / how data is displayed.

<Callout color="primary">
  If you need to implement custom logic data / filtering / sorting for a specific list, this is the place to do it.
</Callout>

Don't worry, we provide a couple starter data provider hooks, which should cover most use cases.

### Static data provider

The `useStaticDataProvider` fetches its data once and filters it using JS-based logic, without any server interaction.

```tsx
import { useStaticDataProvider, type UseDataProviderFn } from "@ubloimmo/uikit";

// fetches a list of pokemon
const fetchPokemon = async (): Promise<Pokemon[]> => {
  // ...
  return [];
}

// create a data provider hook that provides a fetcher
// and forwards the setData callback to the static data provider
const usePokemonData: UseDataProviderFn<Pokemon> = (setData) => {
  return useStaticDataProvider(fetchPokemon, setData);
}
```

## Behavior

Now that we know what our data looks like and have a way to manage it, we can start configuring how the user can interact with it.

Import the `useListConfig` hook to configure a list&apos;s options and behavior.
It yieds a `config` object and various typed methods that mutate it.

> This `config` object is the one and only property supplied to either the `List` component or its related `ListProvider`.

### Barebones Configuration

If all you need to do is display data in a list, this is the simplest way to do it.

```tsx
import { useListConfig } from "@ubloimmo/uikit";

/**
 * Custom hook that configures the list
 */
const usePokemonListConfig = () => {
  // we pass our data provider hook to the useListConfig hook, without calling it
  const { config } = useListConfig<Pokemon>(usePokemonData);

  // and return the config object
  return config;
}
```

## Filtering architecture

When there is a lot of data, it is often useful to allow the user to filter it.

Filtering a list's data revolves around **4 core concepts**.

### Matches

Matches are the building blocks of filtering. They are used to define the criteria for a filter option.

A match can be interpreted as a single condition that must be met for an item to be matched.

It holds three properties:

- `property`: The item's property to match on.
- `comparison`: The comparison operator to use for the match.
- `value`: The value to match against.

### Options

Options are built on top of one or more matches and a user-readable label.

They hold additional properties that define how the option should be displayed and behave.

### Filters

Filters are essentially a labeled collection of one or more options, and offer a way for users to interact (select / unselect) with them.

They also hold additional properties that define how the filter should behave.

### Filter presets

Filter presets also hold a collection of one or more options, but they are meant to be used as a whole.

Once toggled, all options in the preset are selected or unselected.

## Combining multiple filters / options / filter presets

By default, multiple selected options will be combined using the `AND` operator. If a user selects multiple options, the filter will be applied to items that match all of the selected options.

This behavior can be changed by calling the `setOperator()` method returned from the `useListConfig` hook.

## Configuration methods

Matches, options filters and filter presets are, _you guessed it_, built using the `useListConfig` hook and its methods.

> Each method is typed based on the data provider's generic `item` type, ensuring each option is compatible with the list'sdata.

## Sorting architecture

The higher the volume of data displayed in a list, the easier it is for users to lose where they are.
Sorting, along with filtering, naturally becomes a necessity for such users.

### Anatomy of a Sort

A Sort mainly consists of a property, an order and a priority.
Multiple Sorts may be declared and active at the same time, provided they target different `properties`.

```ts
/**
 * The configuration of a single list Sort
 */
type SortConfig<
  TItem extends object,
  TProperty extends FilterProperty<TItem>,
> = {
  /**
   * @required
   */
  readonly property: TProperty;
  /**
   * @default "asc"
   */
  order?: SortOrder<TItem, NoInfer<TProperty>>;
  /**
   * @default 0
   */
  priority?: number;
};
```

#### Property

A Sort can target any `property` of the list item by refering to its path.

Our architecture allows for up to 1 (one) dynamic sorting order per item `property`. We believe multiple sorting configurations per property only serve as a means of losing the user.

```ts
type Item = {
  id: string;
  label: string;
  active?: boolean;
}
```

<Callout color="warning" icon="ExclamationSquareFill">
  If multiple Sorts are targeting the same `property`, the latter will overwrite the former.
</Callout>

#### Order

A Sort's order may be basic or complex.

```ts
/**
 * Sorting order applicable to a list Sort.
 *
 * Either a basic sorting order or a complex one.
 */
export type SortOrder<
  TItem extends object,
  TProperty extends FilterProperty<TItem>,
> = SortOrderBasic | SortOrderComplex<TItem, TProperty>;
```

> Regarless of wether the order is basic or complex, it may be inverted at any time through a user action

##### Basic sorting order

A basic sorting defines whether items should be sorted in ascending or descending order.

```ts
/**
 * Basic sorting order for a single list Sort.
 * Either `asc`: ascending or `desc`: descending
 */
export type SortOrderBasic = "asc" | "desc";
```

Depending on the type of the property the Sort points to, its basic sorting order operates differently:
 - Strings get sorted in alphabetical (A -> Z) order when using `asc`, and in reverse (Z -> A) when using `desc`.
 - Numbers get sorted in ascending (-∞ -> +∞) when using `asc`, and in descending order (+∞ -> -∞) with `desc`.
 - Objects (arrays included) always get sorted after Strings or Numbers, regarless of sorting order.
 - Null values always appear after any non-nullish value, regardless of sorting order.
 - Undefined values get relegated to the end, after even Null values, regardless of sorting order.

<Callout>
In short, when using basic sorting orders, only Strings & Numbers may be sorted in ascending or descending order, while order values follow after.
</Callout>

##### Complex sorting order

Items may be sorted according to a specific, bespoke order.

Let's take for instance an item holding a state. We want items in an `active` state to appear first, followed by pending items, then sort archived items at the end.
This should be represented by the following Sort configuration:

```ts
type State = "archived" | "pending" | "active";
type Item = { id: string; state: State };

const stateSortConfig = {
  property: "state",
  // items where state:active show up first, pending and finally archived
  order: ["active", "pending", "archived"],
} satisfies SortConfig<TItem, "state">;
```

If a property holds a `nullable` value, `null` may also be included in the complex sorting order, grouping null items at the specified position.

Suppose a `null` state means the item is invalid, one could want to show it first to their users.

```ts
type State = "archived" | "pending" | "active";
type Item = { id: string; state: State | null };

const stateSortConfig = {
  property: "state",
  // items where state:null show up first, then active, pending and finally archived
  order: [null, "active", "pending", "archived"],
} satisfies SortConfig<TItem, "state">;
```

> Property values not included in a complex sorting order will always be placed after included ones.

#### Priority

A Sort's `priority` is a number that determines the order in which multiple sorts are applied. Think of it as defining the list's primary, secondary, tertiary sorting order, and so on...

> Only numbers between 0 and positive infinity are valid. Negative properties will get clamped to this range.

<Callout>
 The lower a Sort's `priority` number, the earliest it will be applied.
</Callout>

### Sorting behavior

#### (De)Activation

Sorts may be activated and deactivated at will. **A deactivated sort won't be forwarded to the list's DataProvider** and therefore not influence the order of list items.

A sort's `active` property holds whether it is active or not. Its value may be set during initialization or in runtime.

#### Inversion

A Sort's order (be it basic or complex) may be `inverted` at any point during its lifetime.
When triggered following an interaction, it allows users to easily explore the list's content.

When inverted, basic sorting orders transition from `asc` to `desc` and vice-versa.
Complex sorting orders get reversed.

> Upon inversion, a Sort's `inverted` property gets set to `true`.
> It's `computedOrder` property is also updated accordingly, computed from its initial sorting order and inversion.

#### Prioritization

A Sort's priority may be elevated by calling its `prioritize()` method. Doing so sets its `priority` property to `-1`, thus making it the list's primary sorting order.

> Only one Sort may be prioritized at a time. A Sort remains prioritized as long as no other Sort has been prioritized.

### Sorting methods

- [sort()](?path=/docs/components-list-configuration-methods-sort--docs): Builds and returns a single _sort_ for the list's data.
- [sorts()](?path=/docs/components-list-configuration-methods-sorts--docs): Builds and returns a **map** of _sorts_ for the list's data.

### Visual styling

For users to confidently interact with a list's sorting, they need to be able to identify which Sort they are interacting with.

Each sort holds optional `label` and `iconSet` properties.

#### Label

Either null or a short, human-readable name that identifies which item property the Sort holds.

> It defaults to `null`.

#### Icon Set

A Sort's `iconSet` property influences which icons get displayed while rendered. This is useful for differenciating sorts that handle string number, or any other values from each other.

## Searching via a text query

Text queries are of course also supported by the List component, and may be configured through methods found in the `search` property returned by the `useListConfig()` hook.

This part of the configuration holds properties that define how the text query is handled and passed to the data provider.

### Shape of the search configuration

The search configuration is a plain object with the shape:

```tsx
type ListSearchConfig<TItem extends object> = {
  /**
   * @default null
   */
  initialQuery?: Nullable<string>;
  /**
   * @default []
   */
  properties?: FilterProperty<TItem>[];
  /**
   * @default "contains"
   */
  strategy?: FilterSearchOperator;
  /**
   * @default true
   */
  searchAsOptions?: boolean;
  /**
   * @default 300
   */
  debounceDelay?: number;
}
```

> All search configuration details may be set through dedicated methods (e.g. `search.initialQuery()`), or via a generic setter method (`search.set({ [property]: "value" })`).
> The `search.set()` method allows setting multiple properties at once.

#### Initial query

By default, no text query is set. Should one need to set an initial query, the `initialQuery` property is where you want to look.

```tsx
import { useListConfig } from "@ubloimmo/uikit";

const { search } = useListConfig(usePokemonData);

// both are equivalent
search.initialQuery("my initial query");
search.set({ initialQuery: "my initial query" });
```

#### Search properties

Since a text query only amounts to a string, we need to define which properties of the list's items it should be matched against.

<Callout color="gray" icon={null}>
Allowed properties are based on the list item's shape, and arethe same as the ones used to build filter option matches.
</Callout>

```tsx
import { useListConfig } from "@ubloimmo/uikit";

const { search } = useListConfig(usePokemonData);

// both are equivalent
search.properties("name", "types.0.type.name");
search.set({ properties: ["name", "types.0.type.name"] });
```

<Callout color="warning" icon="ExclamationSquareFill">
At least one property must be set for any text search to be sent to the data provider.
</Callout>

#### Search strategy

The search strategy is used to determine how the text query should be matched against the properties.

It corresponds to a subset of the comparison operators used to build filter option matches.

The following strategies are available:
- `"contains"`: The text query is matched against the properties using the `contains` comparison operator.
- `"startsWith"`: The text query is matched against the properties using the `startsWith` comparison operator.
- `"endsWith"`: The text query is matched against the properties using the `endsWith` comparison operator.

```tsx
import { useListConfig } from "@ubloimmo/uikit";

const { search } = useListConfig(usePokemonData);

// both are equivalent
search.strategy("contains");
search.set({ strategy: "contains" });
```

#### Search as options

By default, the text query and its configuration are parsed and sent as an additional filter to the data provider,
meaning no extra implementation is required to handle it.

For example, given the query `"pikachu"` and the following configuration:

```tsx
{
  properties: ["name", "types.0.type.name"],
  stategy: "contains",
}
```

The data provider's `filter()` method will receive an additional filter with a single option containing one match per property:
```tsx
{
  selectedOptions: [
    {
      // ...rest of the option's data
      matches: [
        {
          property: "name",
          comparison: "contains",
          value: "pikachu",
        },
        {
          property: "types.0.type.name",
          comparison: "contains",
          value: "pikachu",
        },
      ]
    }
  ]
}
```

> Setting `searchAsOptions` to `false` will disable this behavior, and the data provider's `filter()` method's first parameter's `filters` property **will not be hydrated** with the query, only the list's configured options-based fitlers.

##### Consuming the query without `searchAsOptions`

The data provider's `filter()` method's first parameter will **always** contain a `search` property alongside its other data (e.g. `filters`, `filterPresets`, `options`, etc.).
It will be `null` when no valid search configuration was provided (e.g. no properties were set), or an object with the following shape:

```tsx
{
  /**
   * The active, debounced text query, or null if empty
  */
  query: string | null;
  /**
   * The properties the query is to be matched against,
   * as defined in the list's search configuration
  */
  properties: FilterProperty[];
  /**
   * The search strategy used to match the query against the properties,
   * as defined in the list's search configuration
  */
  strategy: "contains" | "startsWith" | "endsWith";
}
```

> Note that all theses properties are not optional, and will always be present as long as `search` isn't null.
> One could easily use them to recreate one or multiple matches if needed.

#### Debouncing

The text query is debounced by `300ms` by default, and the debounce delay can be configured using the `debounceDelay` property.

```tsx
import { useListConfig } from "@ubloimmo/uikit";

const { search } = useListConfig(usePokemonData);

// both are equivalent and set the search debounce delay to 1 second (1000 milliseconds)
search.debounceDelay(1000);
search.set({ debounceDelay: 1000 });
```

> This setting only affects the frequency at which changing the query triggers a new call the the data provider's `filter()` method, not the underlying state's value.

## Search parameters

You might want to show different initial filter options based on the link a user clicked to get to the page.

To do this, you can use the `configureSearchParams()` method. By default, search parameters are not taken into account.

```tsx
import { useListConfig } from "@ubloimmo/uikit";

const { configureSearchParams } = useListConfig(usePokemonData);

useStatic(() => {
  configureSearchParams({ sync: "read" });
});
```

### Synchronization

The `sync` property of the `configureSearchParams()` may be set to:

- `"read"`: The list's options will be read from the search params when the list is mounted.
- `"write"`: The list's options will be written to the search params when the list's options change.
- `"read-write"`: The list's options will be both read from and written to the search params.
- `true`: Equivalent to `"read-write"`.
- `false`: Disables search params synchronization.

### Framework / Router integration

Search params are read and written to the url using the `window.location` property by default.

`configureSearchParams()` accepts both `readParams` and `writeParams` callbacks to override this behavior.

#### react-router-dom example

```tsx
import { useSearchParams } from "react-router-dom";
import { useListConfig, useStatic } from "@ubloimmo/uikit";

const usePokemonListConfig = () => {
  const [searchParams, setSearchParams] = useSearchParams();
  const { configureSearchParams, config } = useListConfig(usePokemonData);

  useStatic(() => {
    configureSearchParams({
      sync: "read-write",
      readParams: () => searchParams,
      writeParams: (params: URLSearchParams) => setSearchParams(params),
    })
  })

  return config;
}
```

    const pikachuOrRaichu = option(

        color: "primary",

        exclusive: true,
      }
    );
