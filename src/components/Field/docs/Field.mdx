## Usage

Render an [input](?path=/docs/components-input-usage--docs), its [label](?path=/docs/components-inputlabel-usage--docs) and [assistive / error text](?path=/docs/components-inputassistivetext-usage--docs) from a single component.

### Input type

By providing a `type` property, a `Field` can render any supported Input and will inherit its specific properties. Defaults to `text` if not provided.

<Canvas of={FieldStories.Types} />

### Common input properties

All common and specific [input properties](?path=/docs/components-input-properties--docs) (e.g. `required`, `disabled`, etc...) are passed directly to the underlying input.

> See [`Input`](?path=/docs/components-input-usage--docs)'s documentation for further details.

### Accessibility

#### Label

Fields use an [`InputLabel`](?path=/docs/components-inputlabel-usage--docs) under the hood.
As such, they inherit both its `label`, `required` and `tooltip` properties and pass them along.

> Clicking on a field's label will focus its input.

#### Assistive text

You might have noticed that a Field's default assistive text is an obvious placeholder.
It is meant to be _replaced_ with an indication to our users about how / why they need to fill the field.
This could be an example, a reason, an explation or any useful text.

> If you really don't want to display an assistive text under a Field's input, set its `assistiveText` property to `null`.

<Canvas of={FieldStories.AssistiveTexts} />

#### Errors & validation

If provided with an `error` property, a `Field` will pass it on to its underlying `Input` and display its `errorText` if provided.

Similarly, if the child `Input` reports an invalid [validity state](https://developer.mozilla.org/en-US/docs/Web/API/ValidityState) (e.g. no value when required and the likes), the parent `Field` will display a corresponding generic error message.

> Only the input's error state will be shown if no `errorText` is provided nor reported.
